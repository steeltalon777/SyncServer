# Анализ архитектуры SyncServer

## 1. Общий обзор системы

**SyncServer** - это backend-система для управления складскими данными, инвентарными операциями, контролем доступа к сайтам и синхронизацией устройств.

### Ключевые характеристики:
- **Авторитетный источник данных** (source of truth) для складского состояния
- **Асинхронный FastAPI** бэкенд на Python
- **PostgreSQL** как основное хранилище данных
- **Токен-базированная аутентификация** для пользователей и устройств
- **Событийно-ориентированная синхронизация** для мобильных устройств

### Целевая аудитория:
1. **Веб-клиенты** (Django admin) - административное управление
2. **Пользовательские клиенты** - работа с инвентарем и операциями
3. **Устройства синхронизации** - мобильные устройства для складских операций

## 2. Архитектурные стили и паттерны

### 2.1. Слоистая архитектура (Layered Architecture)
Проект следует строгой слоистой архитектуре с четким разделением ответственности:

```
Клиенты
    ↓
API слой (FastAPI routes)
    ↓
Сервисный слой (Business Logic)
    ↓
Репозиторный слой (Data Access)
    ↓
Модельный слой (ORM Entities)
    ↓
PostgreSQL
```

### 2.2. Unit of Work паттерн
Используется `UnitOfWork` для управления транзакциями и предоставления единого интерфейса ко всем репозиториям в рамках одной транзакции.

### 2.3. Repository паттерн
Каждая сущность имеет свой репозиторий с методами для доступа к данным.

### 2.4. DTO (Data Transfer Objects)
Используются Pydantic схемы для валидации запросов и ответов.

## 3. Детальный анализ слоев

### 3.1. API слой (`app/api/`)
**Ответственность:**
- Маршрутизация HTTP запросов
- Валидация входных данных (через Pydantic)
- Аутентификация через заголовки (`X-User-Token`, `X-Device-Token`)
- Преобразование исключений в HTTP ответы
- Логирование и отслеживание запросов (middleware)

**Ключевые модули:**
- `routes_*.py` - эндпоинты для различных доменов
- `deps.py` - зависимости FastAPI (DI)
- `exceptions.py` - кастомные исключения

### 3.2. Сервисный слой (`app/services/`)
**Ответственность:**
- Бизнес-правила и инварианты
- Оркестрация workflow
- Валидация бизнес-логики
- Управление транзакциями через UnitOfWork

**Ключевые сервисы:**
- `operations_service.py` - жизненный цикл операций
- `catalog_admin_service.py` - управление каталогом
- `identity_service.py` - управление пользователями
- `access_service.py` - контроль доступа
- `sync_service.py` - синхронизация устройств
- `event_ingest.py` - обработка событий

### 3.3. Репозиторный слой (`app/repos/`)
**Ответственность:**
- Доступ к данным через SQLAlchemy
- Составление запросов
- Базовые CRUD операции
- Оптимизация запросов

**Принципы:**
- Репозитории не содержат бизнес-логики
- Используют async/await для асинхронных операций
- Возвращают ORM модели или DTO

### 3.4. Модельный слой (`app/models/`)
**Ответственность:**
- Определение структуры данных
- Отношения между сущностями
- Ограничения уровня базы данных

**Ключевые модели:**
- `User` и `UserAccessScope` - пользователи и права доступа
- `Site` - складские площадки
- `Category`, `Item`, `Unit` - каталог товаров
- `Operation` и `OperationLine` - складские операции
- `Balance` - инвентарные остатки
- `Device` и `Event` - устройства и события синхронизации

## 4. Модель данных

### 4.1. Основные сущности

#### Пользователи и доступ:
- **User**: UUID идентификатор, токен, глобальная роль (`root`, `chief_storekeeper`, `storekeeper`, `observer`)
- **UserAccessScope**: права доступа к конкретным сайтам (`can_view`, `can_operate`, `can_manage_catalog`)

#### Складские площадки:
- **Site**: физические или логические склады

#### Каталог товаров:
- **Category**: иерархическая структура категорий (adjacency list)
- **Item**: товары с SKU, связь с категориями и единицами измерения
- **Unit**: единицы измерения

#### Операции и остатки:
- **Operation**: документы складских операций (`RECEIVE`, `EXPENSE`, `WRITE_OFF`, `MOVE`, etc.)
- **OperationLine**: строки операций с количеством
- **Balance**: производные остатки (вычисляются из операций)

#### Синхронизация:
- **Device**: зарегистрированные устройства
- **Event**: события синхронизации (иммутабельный лог)

### 4.2. Ключевые архитектурные решения

#### Operation-Driven Inventory:
Остатки (`Balance`) являются производными от операций, а не первичными данными. Это обеспечивает:
- Аудит всех изменений
- Отсутствие прямого редактирования остатков
- Согласованность данных

#### Глобальный каталог:
Каталог товаров общий для всех сайтов, что позволяет:
- Единую номенклатуру
- Легкое копирование товаров между сайтами
- Централизованное управление

#### Site-Scoped Permissions:
Права доступа привязаны к сайтам через `UserAccessScope`, что обеспечивает:
- Гибкое распределение прав
- Изоляцию данных между сайтами
- Масштабируемость

## 5. Бизнес-логика и сервисы

### 5.1. Operations Service
**Сложность:** Высокая
**Ответственность:**
- Валидация типов операций
- Проверка достаточности остатков
- Управление жизненным циклом (`draft` → `submitted` → `cancelled`)
- Обновление балансов

**Ключевые инварианты:**
- Количество не может быть нулевым
- Для операций списания нужны достаточные остатки
- Статусы изменяются по строгому workflow

### 5.2. Catalog Service
**Ответственность:**
- Управление иерархией категорий
- Валидация товаров
- Обеспечение целостности данных

### 5.3. Sync Service
**Ответственность:**
- Идемпотентный прием событий
- Проверка хешей payload
- Упорядочивание событий через `server_seq`

## 6. Аутентификация и авторизация

### 6.1. Механизмы аутентификации:
1. **User Token**: `X-User-Token` для пользователей
2. **Device Token**: `X-Device-Token` для устройств

### 6.2. Уровни доступа:
1. **Root**: глобальный полный доступ
2. **Site-Scoped**: права на конкретных сайтах
3. **Role-Based**: бизнес-роли (`chief_storekeeper`, etc.)

### 6.3. Проверка доступа:
- `access_guard.py` - проверка прав доступа к эндпоинтам
- `access_service.py` - бизнес-логика контроля доступа

## 7. Интеграции и клиенты

### 7.1. Внешние интеграции:
- **PostgreSQL**: основное хранилище
- **Django Admin**: основной административный интерфейс

### 7.2. Клиентские контракты:
- **HTTP REST API** с JSON
- **OpenAPI** документация (`/api/docs`)
- **Стабильные версии** (`/api/v1/`)

### 7.3. Синхронизация устройств:
- Событийный подход
- Идемпотентность операций
- Поддержка офлайн-работы

## 8. Тестирование

### 8.1. Типы тестов:

## 10. Сильные стороны архитектуры

### 10.1. Преимущества:
1. **Четкое разделение ответственности** между слоями
2. **Согласованность данных** через operation-driven inventory
3. **Масштабируемость** через site-scoped permissions
4. **Гибкость** через глобальный каталог
5. **Надежность** через идемпотентную синхронизацию
6. **Тестируемость** через dependency injection

### 10.2. Архитектурные достоинства:
- **ADR (Architecture Decision Records)** - документирование решений
- **Единый источник истины** - избежание расхождений
- **Тонкие контроллеры, толстые сервисы** - правильное распределение логики

## 11. Области для потенциального улучшения

### 11.1. Возможные улучшения:
1. **Кэширование**: Добавить Redis для часто читаемых данных (каталог, балансы)
2. **Фоновая обработка**: Вынести тяжелые операции (отчеты, агрегации) в фоновые задачи
3. **Event Sourcing**: Рассмотреть полный переход к event sourcing для улучшения аудита
4. **GraphQL**: Добавить GraphQL endpoint для сложных запросов клиентов
5. **Метрики и трейсинг**: Добавить OpenTelemetry для распределенной трассировки
6. **Миграции базы данных**: Внедрить Alembic для управления миграциями

### 11.2. Технический долг:
1. **Два access service** (`access_service.py` и `access_service_v2.py`) - нужна консолидация
2. **Отсутствие миграций** - используются скрипты вместо Alembic
3. **Ограниченное покрытие тестами** - некоторые сервисы недостаточно протестированы

## 12. Заключение

SyncServer демонстрирует **зрелую, хорошо спроектированную архитектуру** с четким разделением ответственности и продуманными архитектурными решениями.

**Ключевые успехи:**
- Строгое соблюдение принципов layered architecture
- Эффективное использование Unit of Work для управления транзакциями
- Прозрачная модель данных с operation-driven inventory
- Гибкая система контроля доступа
- Надежная система синхронизации устройств

**Рекомендации для дальнейшего развития:**
1. Консолидировать дублирующийся код (особенно access services)
2. Внедрить систему миграций базы данных
3. Расширить покрытие тестами, особенно интеграционными
4. Рассмотреть внедрение кэширования для повышения производительности

Проект находится в **стабильном состоянии** и готов к дальнейшему развитию с сохранением текущих архитектурных принципов.
