# ТЗ: доработка Django-клиента Warehouse_web под приёмку, выданное имущество и контуры непринятого/утраченного

## 1. Контекст

SyncServer уже поддерживает:
- построчную приёмку для `RECEIVE` и `MOVE`
- регистр `pending_acceptance`
- регистр `lost_assets`
- регистр `issued_assets`
- справочник получателей имущества `recipients`
- операции `ISSUE` и `ISSUE_RETURN`

Django-клиент `Warehouse_web` должен быть доработан как тонкий SSR-клиент поверх новых API SyncServer без дублирования доменной логики на своей стороне.

Ключевой принцип:
- все доменные решения, расчёты остатков, дедупликация получателей и контроль переходов статусов выполняются на стороне SyncServer
- Django хранит только техническое состояние UI, пользовательскую сессию, выбранный активный склад и временные данные формы

## 2. Цель

Доработать Django-клиент так, чтобы пользователь мог:
- создавать и отправлять операции `RECEIVE`, `MOVE`, `ISSUE`, `ISSUE_RETURN`
- выполнять построчную приёмку приходов и перемещений
- видеть и разбирать непринятое имущество
- видеть и разбирать временно утраченное имущество
- выдавать имущество на человека и принимать возврат
- искать, создавать и использовать карточки получателей имущества
- получать список имущества, числящегося на конкретном человеке

## 3. Границы работ

В объём входят:
- клиентский слой `apps/sync_client/*`
- Django views/services/forms/templates для новых сценариев
- серверные интеграционные тесты клиента с моками SyncServer
- обновление навигации и прав в UI

В объём не входят:
- изменение доменной логики SyncServer
- локальное хранение операций, остатков, получателей или регистров имущества в БД Django
- офлайн-режим

## 4. Пользовательские роли и доступ

Роли в UI должны соответствовать ролям SyncServer:
- `storekeeper`
- `chief_storekeeper`
- `observer`
- `root`

Правила доступа:
- `storekeeper` может создавать `RECEIVE`, `MOVE`, `ISSUE`, `ISSUE_RETURN` на доступном складе
- `storekeeper` склада назначения может выполнять приёмку строк по `RECEIVE` и `MOVE`
- `chief_storekeeper` и `root` могут выполнять все операции на всех складах
- разбор `lost_assets` доступен только `chief_storekeeper` и `root`
- `observer` имеет read-only доступ к спискам и карточкам

Клиент не должен реализовывать собственную альтернативную матрицу прав. Он должен:
- скрывать недоступные действия в UI
- дополнительно обрабатывать `403` от SyncServer как источник истины

## 5. Новые пользовательские разделы

В клиенте должны появиться разделы меню:
- `Операции`
- `Приёмка`
- `Непринятое`
- `Утраченное`
- `Выданное имущество`
- `Получатели имущества`

Минимальная навигация:
- список операций
- карточка операции
- экран приёмки операции
- список непринятого имущества
- список утраченного имущества
- список выданного имущества
- список/поиск получателей
- карточка получателя с табом "Что на нём числится"

## 6. Требования к клиентскому слою SyncServer

В `apps/sync_client/` необходимо добавить новые API-клиенты или расширить существующие:

### 6.1. Operations API
- `POST /api/v1/operations`
- `GET /api/v1/operations`
- `GET /api/v1/operations/{id}`
- `PATCH /api/v1/operations/{id}`
- `POST /api/v1/operations/{id}/submit`
- `POST /api/v1/operations/{id}/cancel`
- `POST /api/v1/operations/{id}/accept-lines`

### 6.2. Assets API
- `GET /api/v1/pending-acceptance`
- `GET /api/v1/lost-assets`
- `POST /api/v1/lost-assets/{operation_line_id}/resolve`
- `GET /api/v1/issued-assets`

### 6.3. Recipients API
- `GET /api/v1/recipients`
- `POST /api/v1/recipients`
- `POST /api/v1/recipients/merge`

### 6.4. Общие требования к sync_client
- единый обработчик ошибок `401/403/404/409/422`
- единый DTO-слой для новых ответов
- логирование request id и endpoint при ошибках
- нормальная обработка `409` как бизнес-конфликта, а не "падение системы"

## 7. Функциональные требования по экранам

### 7.1. Список операций

Доработать список операций:
- показывать тип операции
- показывать `status`
- показывать `acceptance_state`
- показывать склад-источник и склад-назначение для `MOVE`
- показывать получателя для `ISSUE` и `ISSUE_RETURN`
- выводить быстрые действия:
  - `Открыть`
  - `Отправить`
  - `Отменить`
  - `Принять`, если операция требует приёмки и пользователь имеет право

Фильтры:
- склад
- тип операции
- статус
- `acceptance_state`
- поиск по заметке, получателю, id операции

### 7.2. Карточка операции

Карточка операции должна показывать:
- шапку операции
- строки с колонками `qty`, `accepted_qty`, `lost_qty`, `pending_qty`
- текущий `acceptance_state`
- блок истории действий по приёмке и разбору утраты, если API будет расширен под это чтение

Для `MOVE` и `RECEIVE` при `acceptance_required=true` должно быть видно:
- что операция ещё не принята
- сколько осталось принять
- есть ли строки с утратой

### 7.3. Форма создания операции RECEIVE

Форма должна:
- позволять создать документ `RECEIVE`
- поддерживать несколько строк
- после отправки вести пользователя в карточку операции

После `submit` клиент должен:
- показывать, что документ перешёл в ожидание приёмки
- не трактовать его как уже принятый складской приход

### 7.4. Форма создания операции MOVE

Форма должна:
- выбирать склад-источник
- выбирать склад-назначение
- не требовать прав на склад-назначение для создания, если отправитель работает на складе-источнике
- показывать пользователю, что после отправки имущество попадёт в ожидание приёмки на складе назначения

### 7.5. Экран приёмки

Новый экран `Приёмка` должен работать по операции `RECEIVE` или `MOVE`.

Требования:
- список строк операции
- по каждой строке поля:
  - `Принято`
  - `Утрачено`
  - `Комментарий`
- автоматически показывать остаток `pending_qty`
- валидировать на клиенте, что `accepted_qty + lost_qty` не превышает остаток по строке
- отправлять единый payload в `POST /operations/{id}/accept-lines`
- после успешного ответа обновлять экран без полной потери контекста формы

UX-требования:
- приёмка должна быть удобна для работы по нескольким строкам подряд
- по умолчанию курсор должен помогать быстрому вводу количества
- для полностью доставленной строки должен быть быстрый сценарий "принять всё"

### 7.6. Список непринятого имущества

Новый раздел `Непринятое`.

Требования:
- получать данные из `GET /pending-acceptance`
- показывать:
  - операция
  - строка операции
  - склад назначения
  - склад-источник
  - номенклатура
  - количество
  - дата обновления
- из строки списка должен быть переход в карточку операции и экран приёмки

### 7.7. Список утраченного имущества

Новый раздел `Утраченное`.

Требования:
- получать данные из `GET /lost-assets`
- показывать:
  - операция
  - строка операции
  - склад назначения
  - склад-источник
  - номенклатура
  - количество
- для `chief_storekeeper/root` доступны действия:
  - `Найдено и принять на склад назначения`
  - `Вернуть на склад-источник`
  - `Списать`

Для каждого действия должен открываться отдельный confirm-flow с выбором количества и комментарием.

### 7.8. Форма выдачи имущества

Форма `ISSUE` должна:
- выбирать склад
- выбирать номенклатуру и количество
- выбирать получателя

Выбор получателя:
- основной сценарий через autocomplete по `GET /recipients`
- если ничего не найдено, предложить быстрое создание карточки через `POST /recipients`
- после создания сразу подставлять нового получателя в форму

В форме нужно исключить ручную работу с `recipient_id` и UUID. Только человекочитаемый поиск и выбор.

### 7.9. Форма возврата имущества

Форма `ISSUE_RETURN` должна:
- позволять выбрать получателя
- после выбора получателя показывать, что на нём числится
- позволять быстро выбрать позицию из `issued-assets`
- не заставлять пользователя вручную вспоминать, что было выдано

### 7.10. Список выданного имущества

Раздел `Выданное имущество` должен:
- получать данные из `GET /issued-assets`
- поддерживать фильтры по получателю, номенклатуре, поиску
- использоваться как источник для:
  - возврата имущества
  - контроля имущества на человеке
  - сценария увольнения

### 7.11. Получатели имущества

Раздел `Получатели имущества` должен:
- показывать список карточек
- поддерживать поиск
- открывать карточку получателя

Карточка получателя должна содержать:
- основные данные
- список имущества, числящегося на человеке
- историю операций выдачи/возврата, если такая выборка будет добавлена позднее

Для `chief_storekeeper/root` предусмотреть UI для merge дублей:
- выбор source
- выбор target
- подтверждение объединения

## 8. Формы и валидация

На стороне Django-клиента должна быть только UI-валидация:
- обязательные поля формы
- числа больше нуля
- суммы по строке при приёмке не превышают остаток
- нельзя отправить пустую форму

Запрещено:
- пересчитывать остатки локально
- самостоятельно дедуплицировать получателей
- самостоятельно определять, можно ли списать утрату

Финальная валидация всегда приходит от SyncServer.

## 10. Обработка ошибок в UI

Клиент должен различать:
- `403` — нет прав
- `404` — объект не найден
- `409` — бизнес-конфликт
- `422` — ошибка ввода
- `5xx` — недоступность сервиса

Требования к UX:
- ошибки `409` и `422` показывать рядом с формой понятным текстом
- после ошибки не терять введённые пользователем данные формы
- для `403` выводить понятное сообщение без stack trace
- для `5xx` показывать сервисное сообщение и возможность повторить действие

## 11. Нефункциональные требования

- Django-клиент не должен хранить копии регистров имущества в своей БД
- все списки должны поддерживать пагинацию
- страницы списков должны быть пригодны для работы без SPA
- на всех действиях изменения состояния должен быть CSRF-protection и server-side validation
- код интеграции должен идти через единый слой `apps/sync_client`, без прямых вызовов `httpx/requests` из views

## 13. Критерии приёмки

Функционал считается реализованным, если:
- кладовщик склада-источника может создать `MOVE`
- кладовщик склада-назначения может принять строки этого `MOVE`
- до приёмки имущество видно в разделе `Непринятое`, а не в складском остатке назначения
- при недостаче имущество попадает в раздел `Утраченное`
- `chief_storekeeper/root` могут разобрать утрату тремя действиями
- кладовщик может выдать имущество на человека через поиск/автосоздание получателя
- по карточке получателя видно, что на нём числится
- возврат имущества использует данные из регистра `issued-assets`
- все новые действия сопровождаются корректной обработкой `403/409/422`

## 15. Ожидаемый результат

После завершения работ Django-клиент должен покрывать полный пользовательский контур:
- приход/перемещение
- построчная приёмка
- непринятое и временно утраченное имущество
- выдача имущества на человека
- возврат имущества
- просмотр имущества, числящегося на человеке

При этом вся доменная логика остаётся централизованной в SyncServer, а Django-клиент остаётся тонким управляемым интерфейсом.
