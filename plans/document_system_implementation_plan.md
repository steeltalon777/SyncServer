# План внедрения системы документов в SyncServer

## Контекст
SyncServer уже имеет сущность операций ([`Operation`](app/models/operation.py:25)) как факты движения товаров, но печатные документы (накладные, акты, акты приемки) не выделены в отдельную модель. Клиенты должны уметь показывать и печатать документы, но сейчас для этого нужно собирать данные из живых справочников, что приводит к проблемам при их изменении.

Цель этого плана — добавить в SyncServer отдельный контур документов, который:
1. Хранит неизменяемые слепки печатных данных.
2. Обеспечивает стабильную печать даже после изменений справочников.
3. Позволяет клиентам получать документы через отдельный API.
4. Поддерживает разные типы документов (накладные, акты и т.д.).
5. Интегрируется с существующим UoW и архитектурой.

## Архитектурные принципы
1. **Документ — это неизменяемый слепок**. После финализации его payload не меняется.
2. **Связь с исходной сущностью**. Документ привязан к источнику (например, операция) через отдельную таблицу связей, сохраняя ссылочную целостность.
3. **Отдельный API контур**. Основной доступ к документам происходит через `/documents/...`, а не через загрузку полного payload внутри операций.
4. **Payload включает все печатные реквизиты**. Не только ID, а полные текстовые слепки заголовков, сторон, строк, подписей.
5. **Плавная эволюция**. Начинаем с документов на основе операций, но оставляем возможность добавлять документы для других источников (отчёты, акты инвентаризации).

## Фаза 1: Базовая модель данных и репозитории (2–3 недели)
**Цель:** Создать таблицы документов и связи, добавить репозиторий в [`UnitOfWork`](app/services/uow.py:19).

### 1.1. Миграция базы данных
- **Задача:** Добавить таблицы `documents` и `document_operations` по образцу из [`plans/syncserver_enhancements_plan.md`](plans/syncserver_enhancements_plan.md:138), но с уточнениями.
- **Действия:**
  - Подготовить SQL миграцию (Alembic) со следующими полями:
    - `id` UUID PRIMARY KEY
    - `document_type` VARCHAR(50) NOT NULL (например, 'waybill', 'acceptance_certificate')
    - `document_number` VARCHAR(100) — уникальный номер документа, назначаемый сервером по сквозной нумерации внутри типа и площадки.
    - `revision` INTEGER NOT NULL DEFAULT 0
    - `status` VARCHAR(32) NOT NULL ('draft', 'finalized', 'void', 'superseded')
    - `site_id` INTEGER REFERENCES sites(id) — площадка, к которой относится документ.
    - `template_name` VARCHAR(100) — идентификатор шаблона, использованного при генерации.
    - `template_version` VARCHAR(32) — версия шаблона.
    - `payload_schema_version` VARCHAR(32) — версия схемы payload.
    - `payload` JSONB NOT NULL — самодостаточный слепок данных для печати.
    - `payload_hash` VARCHAR(64) — хэш payload для контроля неизменности.
    - `created_by_user_id` UUID REFERENCES users(id)
    - `created_at` TIMESTAMPTZ NOT NULL DEFAULT now()
    - `finalized_at` TIMESTAMPTZ
    - `supersedes_document_id` UUID REFERENCES documents(id) — для связи с предыдущей ревизией.
  - Таблица `document_operations`:
    - `document_id` UUID NOT NULL REFERENCES documents(id) ON DELETE CASCADE
    - `operation_id` UUID NOT NULL REFERENCES operations(id) ON DELETE CASCADE
    - PRIMARY KEY (document_id, operation_id)
  - Создать индексы: по `document_type`, `document_number`, `status`, `site_id`, `created_at`, `operation_id`.
- **Критерии успеха:** Миграция применяется без ошибок, таблицы доступны в БД.

## Фаза 2: Сервис формирования документов и базовый API (3–4 недели)
**Цель:** Реализовать сервис, который формирует документ из операции, и предоставить минимальный API для работы с документами.

### 2.1. Сервис формирования документов
- **Задача:** Создать [`DocumentService`](app/services/document_service.py) с методом `generate_from_operation`.
- **Действия:**
  - Реализовать логику сборки payload накладной:
    - Взять операцию и её строки.
    - Добавить исторические снапшоты из [`Operation.recipient_name_snapshot`](app/models/operation.py:70) и [`OperationLine.item_name_snapshot`](app/models/operation.py:225) и др.
    - Добавить слепки площадок (название, адрес, реквизиты) — для этого нужен запрос к [`SitesRepo`](app/repos/sites_repo.py).
    - Добавить данные ответственных лиц (слепки из [`User`](app/models/user.py)).
    - Добавить номер документа согласно сквозной нумерации (можно использовать sequence per site/type).
    - Определить шаблон по умолчанию (например, `waybill_v1`).
  - Метод должен возвращать объект `Document` в статусе `draft` или сразу `finalized` (в зависимости от политики).
  - Проверить, что payload включает все необходимые для печати поля.
- **Критерии успеха:** Тест создаёт документ по операции, payload содержит все ожидаемые поля.

### 2.2. Эндпоинты документов
- **Задача:** Добавить роутер [`routes_documents.py`](app/api/routes_documents.py) с основными операциями.
- **Действия:**
  - `POST /documents/generate` — генерация документа из указанной операции (с параметрами типа, шаблона).
  - `GET /documents/{document_id}` — получение полного документа (метаданные + payload).
  - `GET /documents` — список документов с фильтрацией по типу, статусу, площадке, дате.
  - `PATCH /documents/{document_id}/status` — изменение статуса (например, финализация, аннулирование).
  - `GET /operations/{operation_id}/documents` — список документов по операции (только метаданные, без payload).
  - `POST /operations/{operation_id}/documents` — удобный shortcut для генерации документа определённого типа.
  - Все эндпоинты должны учитывать права доступа по площадке.
- **Критерии успеха:** Все эндпоинты работают, возвращают корректные коды и данные.

### 2.3. Интеграция с жизненным циклом операции
- **Задача:** Автоматически создавать документ при финализации операции (например, при submit).
- **Действия:**
  - В [`OperationsService.submit_operation()`](app/services/operations_service.py) (или в соответствующем месте) вызывать `DocumentService.generate_from_operation` с параметром `auto_finalize=True`.
  - Либо добавить фоновую задачу (отложенную), чтобы не замедлять submit.
  - Предусмотреть возможность отключения автогенерации через конфиг.
- **Критерии успеха:** При отправке операции создаётся документ в статусе `finalized`.

### 2.4. Базовые тесты
- **Задача:** Покрыть сервис и API юнит- и интеграционными тестами.
- **Действия:**
  - Написать `tests/test_document_service.py`.
  - Написать `tests/test_documents_routes.py`.
  - Проверить сценарии: генерация, финализация, аннулирование, повторная генерация.
- **Критерии успеха:** Прохождение всех тестов с coverage >80% по добавленному коду.

## Фаза 3: Клиентская печать и улучшения payload (2–3 недели)
**Цель:** Дать клиентам возможность получать документ в форматах, удобных для печати, и доработать payload.

### 3.1. Рендеринг HTML/PDF
- **Задача:** Добавить возможность отдавать документ в виде HTML или PDF.
- **Действия:**
  - Создать шаблоны Jinja2 для каждого типа документа (например, `templates/documents/waybill.html`).
  - Реализовать `DocumentRenderer`, который принимает payload и шаблон, генерирует HTML.
  - Интегрировать библиотеку для конвертации HTML в PDF (WeasyPrint или аналогичную).
  - Добавить эндпоинты:
    - `GET /documents/{document_id}/render?format=html`
    - `GET /documents/{document_id}/render?format=pdf`
  - Кэшировать сгенерированный PDF/HTML на короткое время.
- **Критерии успеха:** Клиент может запросить PDF и получить файл, идентичный визуально макету накладной.

### 3.2. Расширение payload дополнительными реквизитами
- **Задача:** Включить в payload все печатные реквизиты, которых сейчас не хватает.
- **Действия:**
  - Добавить слепки организационных реквизитов площадки (юридическое название, адрес, ИНН, контакты).
  - Добавить поля основания документа (приказ, договор, заявка).
  - Добавить подписи с ролями (например, «Сдал», «Принял», «Главный бухгалтер») — пока как текстовые заглушки.
  - Добавить локализационные настройки (формат даты, разделители чисел, валюта).
- **Критерии успеха:** Payload становится самодостаточным для печати типового бланка накладной.

### 3.3. Поддержка многоязычности (опционально)
- **Задача:** Добавить возможность формировать документ на разных языках.
- **Действия:**
  - В payload включать язык.
  - Шаблоны иметь в нескольких языковых версиях.
  - При генерации указывать язык через параметр.
- **Критерии успеха:** Можно сгенерировать одну и ту же накладную на русском и английском.

### 3.4. Документация API для клиентов
- **Задача:** Описать в [`API_REFERENCE.md`](docs/API_REFERENCE.md) новые эндпоинты и форматы.
- **Действия:**
  - Добавить раздел «Документы».
  - Привести примеры запросов и ответов.
  - Объяснить жизненный цикл документа.
- **Критерии успеха:** Разработчики клиентов могут интегрироваться с документами, используя только документацию.

## Фаза 4: Улучшения и поддержка других типов документов (3–4 недели)
**Цель:** Расширить систему на другие типы документов и добавить продвинутые функции.

### 4.1. Универсальные источники документов

- **Действия:**
  - Добавить таблицу `document_sources` с полями `source_type` (например, 'operation', 'report', 'inventory') и `source_id` (UUID или INTEGER).
  - Переделать `document_operations` в `document_sources` (или оставить обе для обратной совместимости).
  - Адаптировать репозиторий и сервис для работы с любым source_type.
- **Критерии успеха:** Можно привязать документ к отчёту или акту инвентаризации.

### 4.2. Шаблоны документов в БД
- **Задача:** Перенести хранение шаблонов из файловой системы в БД для динамического управления.
- **Действия:**
  - Создать таблицу `document_templates` с полями `name`, `version`, `document_type`, `content` (Jinja2 HTML), `settings`.
  - Добавить CRUD API для шаблонов (только для администраторов).
  - Модифицировать `DocumentRenderer` для загрузки шаблона из БД.
- **Критерии успеха:** Администратор может загрузить новый шаблон через API без деплоя кода.

### 4.3. Аудит печати и скачивания
- **Задача:** Вести журнал, кто и когда запрашивал печать документа.
- **Действия:**

  - Логировать вызовы эндпоинтов рендеринга и скачивания.
  - Добавить эндпоинт для просмотра аудита по документу (для администраторов).
- **Критерии успеха:** В админке видна история обращений к каждому документу.

### 4.4. Массовая генерация и очередь задач

- **Действия:**
  - Интегрировать с Celery/Redis или использовать async задачи внутри приложения.
  - Добавить эндпоинт `POST /documents/batch-generate` с списком operation_id.
  - Возвращать идентификатор задачи и позволять запрашивать статус.
- **Критерии успеха:** Пользователь может запустить генерацию 100+ документов без блокировки интерфейса.
