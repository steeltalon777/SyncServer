# ТЗ на разработку клиентов для Sync API

Документ задает требования к клиентским приложениям (мобильным/desktop/POS/агентам), которые интегрируются с данным Sync API.

## 1. Цель

Разработать клиент синхронизации, который:

- стабильно работает при нестабильной сети,
- не теряет события,
- корректно обрабатывает дубликаты и коллизии,
- выполняет инкрементальную синхронизацию событий и справочников,
- соблюдает требования безопасности (токен устройства, заголовки).

---

## 2. Термины

- **Сайт (`site_id`)** — площадка/торговая точка.
- **Устройство (`device_id`)** — конкретный клиентский инстанс.
- **Токен устройства (`X-Device-Token`)** — обязательный секрет для авторизации.
- **Событие (`event`)** — бизнес-факт, отправляемый на сервер.
- **`event_uuid`** — уникальный ID события (идемпотентность).
- **`server_seq`** — серверный монотонный порядковый номер событий (для `/pull`).
- **Outbox** — локальная очередь неотправленных событий.

---

## 3. Поддерживаемые endpoints

## 3.1 `POST /ping`

Назначение:

- heartbeat,
- проверка доступа,
- получение `server_seq_upto`.

Тело запроса:

```json
{
  "site_id": "uuid",
  "device_id": "uuid",
  "last_server_seq": 123,
  "outbox_count": 7,
  "client_time": "2026-01-01T10:00:00Z"
}
```

Обязательный заголовок:

- `X-Device-Token: <token>`

Опциональный заголовок:

- `X-Client-Version: <version>`

Ответ:

- `server_time`
- `server_seq_upto`
- `backoff_seconds`

## 3.2 `POST /push`

Назначение: отправка пачки событий outbox.

Тело запроса:

```json
{
  "site_id": "uuid",
  "device_id": "uuid",
  "batch_id": "uuid",
  "events": [
    {
      "event_uuid": "uuid",
      "event_type": "writeoff",
      "event_datetime": "2026-01-01T10:00:00Z",
      "schema_version": 1,
      "payload": {
        "doc_id": "DOC-1",
        "doc_type": "writeoff",
        "comment": "reason",
        "lines": [
          { "item_id": "uuid", "qty": "1.500", "batch": "B1" }
        ]
      }
    }
  ]
}
```

Ограничение:

- `events.length <= MAX_PUSH_EVENTS` (на текущем сервере default 500).

Ответ:

- `accepted[]` — принятые новые события,
- `duplicates[]` — уже существующие с тем же payload,
- `rejected[]` — отклоненные (например, `uuid_collision`),
- `server_seq_upto`, `server_time`.

## 3.3 `POST /pull`

Назначение: получить события сервера после курсора `since_seq`.

Тело запроса:

```json
{
  "site_id": "uuid",
  "device_id": "uuid",
  "since_seq": 120,
  "limit": 200
}
```

Ответ:

- `events[]` (в порядке `server_seq`),
- `next_since_seq`,
- `server_seq_upto`,
- `server_time`.

## 3.4 `POST /catalog/items` и `POST /catalog/categories`

Назначение: инкрементальная синхронизация справочников.

Тело:

```json
{
  "updated_after": "2026-01-01T00:00:00Z",
  "limit": 100
}
```

Обязательные заголовки:

- `X-Site-Id`
- `X-Device-Id`
- `X-Device-Token`

Опционально:

- `X-Client-Version`

Ответ:

- список сущностей,
- `next_updated_after` — курсор следующей страницы,
- `server_time`.

---

## 4. Обязательные требования к клиенту

## 4.1 Идентификация и секреты

Клиент **должен**:

- безопасно хранить `device_id`, `site_id`, `device_token` в защищенном локальном хранилище,
- передавать `X-Device-Token` на `/ping`, `/push`, `/pull`,
- передавать `X-Site-Id`, `X-Device-Id`, `X-Device-Token` на `/catalog/*`,
- уметь обновлять `X-Client-Version` при релизе клиента.

## 4.2 Outbox и идемпотентность

Клиент **должен**:

- присваивать каждому исходящему событию новый `event_uuid` (UUIDv4),
- хранить событие в outbox до подтверждения от `/push`,
- удалять из outbox элементы из `accepted[]` и `duplicates[]`,
- переводить `rejected.uuid_collision` в отдельный статус «конфликт» для ручной/авто-обработки,
- избегать локальной генерации двух разных payload с одним `event_uuid`.

## 4.3 Cursor-based pull

Клиент **должен**:

- хранить `since_seq` отдельно для каждого `site_id`,
- обновлять `since_seq = next_since_seq` только после успешной локальной обработки полученных `events[]`,
- продолжать pull страницами, пока:
  - есть новые события, или
  - `next_since_seq < server_seq_upto`.

## 4.4 Catalog sync

Клиент **должен**:

- хранить курсор `updated_after` отдельно для `items` и `categories`,
- обновлять курсор только после атомарного применения страницы в локальное хранилище,
- поддерживать `is_active=false` как мягкое удаление/деактивацию записи.

## 4.5 Retry и backoff

Клиент **должен**:

- применять retry для сетевых/5xx ошибок,
- учитывать 429 (rate limit) и выдерживать паузу,
- для `/ping` — интервал не чаще 1 раза в 5 секунд,
- для `/push` — не чаще 1 запроса в секунду на устройство,
- использовать exponential backoff с jitter.

Рекомендуемый backoff:

- `base = 1s`, `factor = 2`, `max = 60s`, `jitter = ±20%`.

## 4.6 Консистентность времени и форматов

Клиент **должен**:

- отправлять `datetime` в ISO-8601 (UTC, с `Z`),
- отправлять `qty` как decimal-строку с точностью до 3 знаков,
- соблюдать schema_version текущего события.

---

## 5. Протокол обработки ответов `/push`

Для каждого `event_uuid` из ответа:

1. Если в `accepted[]`:
   - отметить как доставлено,
   - сохранить `server_seq`.
2. Если в `duplicates[]`:
   - считать доставленным ранее,
   - убрать из outbox.
3. Если в `rejected[]`:
   - сохранить причину,
   - не ретраить бесконечно один и тот же конфликт,
   - поднять в UI/журнал.

---

## 6. Нефункциональные требования

- Локальное хранилище должно переживать перезапуск приложения.
- Все операции синхронизации должны быть устойчивы к внезапному завершению процесса.
- Логи клиента должны включать:
  - `request_id` из ответа сервера,
  - endpoint,
  - latency,
  - размер батча,
  - статус/код ошибки.
- Клиент должен поддерживать работу в режиме «offline-first».

---

## 7. Безопасность

- Использовать только HTTPS в production.
- Не логировать `X-Device-Token` в открытом виде.
- Токен хранить в secure storage (Keychain/Keystore/DPAPI и т.п.).
- Реализовать механизм удаленной инвалидизации устройства (в рамках продуктового слоя).

---

## 8. Минимальный алгоритм синхронизации (референс)

1. Запуск клиента:
   - загрузить `site_id/device_id/token/since_seq/catalog cursors`.
2. `POST /ping`.
3. Если в outbox есть события:
   - отправлять батчами `POST /push` (до лимита).
4. Выполнять `POST /pull` страницами до догонки `server_seq_upto`.
5. Выполнять `/catalog/items` и `/catalog/categories` страницами по `updated_after`.
6. Засыпать до следующего sync-цикла.

---

## 9. Критерии приемки клиента

Клиент считается готовым, если:

- корректно проходит happy-path для `/ping`, `/push`, `/pull`, `/catalog/*`;
- корректно обрабатывает `403`, `422`, `429`, `500`;
- не теряет события при падении процесса между ретраями;
- корректно обрабатывает `duplicate` и `uuid_collision`;
- выдерживает длительный офлайн-режим и последующую догонку;
- все синхронизационные операции покрыты интеграционными тестами клиента.

---

## 10. Что можно расширить в следующей версии ТЗ

- Версионирование API (`/v1`).
- Контракт массового ACK/receipt.
- CDC/stream-подход вместо polling pull.
- Централизованный rate-limit (Redis).
- Полноценный механизм пере-регистрации/ротации токена устройства.
